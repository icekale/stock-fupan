# A 股每日复盘系统设计文档

日期：2026-05-26  
状态：已确认设计草案，等待用户复核后进入实施计划  
目标版本：v0.1 收盘复盘闭环

## 1. 项目目标

构建一个面向 A 股市场的每日复盘程序。系统先以本地网页应用形式运行，后续扩展为定时生成和推送的复盘工作台。

核心目标是：在每天收盘后自动采集行情和新闻，识别市场主线，生成可信、可编辑、可导出的复盘报告。报告视觉风格参考用户提供的移动端长图：深色头图、白底章节、板块卡片、清晰的结论和观察清单。

v0.1 的目标不是一次性覆盖所有自动化能力，而是跑通一条真实可用的收盘复盘闭环：行情采集、主线评分、新闻归因、LLM 文案、事实校验、人工修正、HTML/PNG 导出、快照留档。

## 2. 已确认决策

### 2.1 产品范围

- 市场范围：A 股全市场。
- 复盘风格：全市场数据底座，输出偏短线主线复盘。
- 运行方式：先本地网页应用，后续服务器定时任务。
- 报告源格式：HTML 是唯一源格式，导出 PNG、Markdown，PDF 后续增强。
- 编辑模式：事实数据锁定，人工备注和修正说明独立保存。
- 报告模板：桌面编辑模板和移动端长图模板共用同一份 `ReportDTO`。
- 审核流程：生成后必须人工审核，审核通过后再导出。

### 2.2 数据与模型

- 行情数据：v0.1 默认 AkShare，预留 TickFlow、TuShare 等 provider adapter。
- 新闻搜索：Anspire API Keys，采用全网检索加财经可信站点加权。
- 新闻保存：保存标题、链接、来源、摘要和 Anspire 原始 JSON，不保存网页全文。
- LLM：默认 OpenAI，备选 DeepSeek、通义等 OpenAI-compatible 接口。
- LLM 输出：结构化 JSON，必须经过事实校验。
- LLM 记录：保存 prompt、模型、参数、输出 JSON、校验结果。

### 2.3 技术栈

- 仓库结构：monorepo。
- 前端：Next.js、React、TypeScript、Tailwind、shadcn/ui。
- 后端：Python FastAPI。
- 类型同步：由 FastAPI OpenAPI schema 生成前端类型。
- 包管理：pnpm + uv。
- 存储：SQLite + 本地文件目录。
- 导出 PNG：Playwright 截图 HTML。
- 启动方式：开发脚本 + Docker Compose 一键启动。

## 3. 系统架构

系统分为六个主要模块：

1. 前端编辑应用：负责报告生成入口、任务进度、人工修正、预览和导出。
2. 后端编排服务：负责任务创建、行情采集、新闻搜索、评分、LLM 生成、校验和资产管理。
3. 数据 provider 层：将 AkShare、Anspire、OpenAI 等外部能力封装为可替换 adapter。
4. 规则引擎：计算市场状态、板块强度、个股候选、风险提示。
5. 报告渲染层：根据 `ReportDTO` 渲染桌面审核页面和移动长图 HTML。
6. 本地持久化层：SQLite 保存结构化状态，本地目录保存快照和导出物。

### 3.1 数据流

1. 用户在网页中选择交易日和报告类型，v0.1 只支持收盘复盘。
2. 前端创建生成任务。
3. 后端采集 AkShare 行情数据，生成市场事实快照。
4. 规则引擎计算市场状态标签和板块评分。
5. 后端调用 Anspire 搜索新闻，按财经可信来源加权并生成板块催化候选。
6. 后端组装 `ReportDTO`，调用 OpenAI 生成结构化文案 JSON。
7. 事实校验器检查 LLM 输出中的数字、股票名、板块名、涨跌幅和成交额。
8. 校验失败时最多自动重试 3 次；仍失败则标红，并允许用户选择重试、规则模板或手动编辑。
9. 用户在前端审核报告，添加人工备注、调整标签或排序。
10. 用户确认后，系统渲染 HTML 并用 Playwright 导出 PNG。
11. SQLite 记录报告索引和任务状态，本地目录保存完整版本资产。

## 5. 存储设计

### 5.1 SQLite 核心表

`reports`

- 报告索引表。
- 字段包含交易日、报告类型、版本号、状态、资产目录、算法版本、创建时间、更新时间。

`report_tasks`

- 后台任务表。
- 字段包含任务类型、状态、当前阶段、触发来源、错误信息、重试次数、日志摘要、关联报告版本。

`market_facts`

- 核心行情摘要和数据血缘表。
- 字段包含指标名、指标值、provider、接口名、采集时间、原始快照路径。

`sector_scores`

- 板块评分表。
- 字段包含板块名、总分、排名、因子分、新闻分、资金流置信度、算法版本。

`overrides`

- 人工修正表。
- 字段包含报告版本、目标类型、目标 ID、操作类型、修正内容、创建时间。
- 人工修正永不覆盖原始事实，只影响最终展示和可选的 LLM 重写。

v0.2 再增加自选股、交易日历和调度相关表。

### 5.2 报告资产目录

每次生成或重跑都新建版本：

```text
reports/
  2026-05-26/
    close/
      v001/
        snapshot.json
        facts.json
        news_raw.json
        llm_calls.json
        report.dto.json
        report.html
        report.png
        notes.json
      v002/
        ...
    noon/
      v001/
        ...
```

v0.1 只实现 `close` 类型。v0.2 增加 `noon`。v0.3 增加 Markdown 导出和版本保留策略。

## 6. 规则评分设计

### 6.1 主线评分

主线评分采用综合评分，短线强度权重较高。输入因子包括：

- 涨停数、连板、20cm、封板质量。
- 板块涨幅、上涨占比、板块内个股扩散。
- 成交额、放量、资金流。
- 涨跌比和市场情绪。
- 新闻催化强度。

v0.1 使用 `sector_score_v1` 算法版本。每份报告快照记录当次评分权重和算法版本。

### 6.2 午盘和收盘权重

系统预留午盘和收盘两套权重。

- 午盘更重即时强度、异动速度、上午成交量能和扩散。
- 收盘更重全天确认、涨停连板、成交资金、板块涨跌比。

v0.1 只实现收盘权重。v0.2 增加午盘权重。

### 6.3 资金轮动

资金轮动后续采用板块评分变化和资金流数据双轨判断。若资金流缺失，则使用成交额变化替代，并明确标注低置信度。

资金轮动作为 v0.2 或 v0.3 增强项，不阻塞 v0.1。

### 6.4 市场状态标签

第一版支持简单市场状态标签：

- 普涨
- 普跌
- 分化
- 缩量
- 放量
- 高热分歧
- 情绪退潮

状态标签影响 LLM 文案语气和模块排序。例如高热分歧时风险提示前置，普涨放量时强调主线扩散。

## 7. 新闻归因设计

v0.1 默认使用 Anspire API。

策略：

1. 对强势板块和核心关键词进行全网搜索。
2. 对财经可信站点加权，包括东方财富、财联社、证券时报、上证报、交易所公告等。
3. 每个板块形成若干候选催化摘要。
4. 编辑页显示完整来源信息。
5. 长图默认只显示摘要，是否显示来源由配置控制。

保存粒度：

- 结构化字段：标题、链接、来源、摘要、发布时间、匹配板块。
- 原始快照：Anspire 返回 JSON。
- 不抓取和保存网页全文。

## 8. LLM 生成与事实校验

### 8.1 生成原则

LLM 只基于结构化事实、新闻摘要和人工备注生成文案，不负责发明事实或排序。

默认语言风格：短线复盘口吻，直接、克制、关注主线强弱和风险。后续支持风格模板切换。

### 8.2 输出结构

LLM 输出结构化 JSON，包含：

- 今日结论。
- 盘面总览。
- 强势板块点评。
- 个股关注。
- 明日观察。
- 风险提示。

### 8.3 校验策略

事实校验器检查：

- 指数、成交额、涨停数、跌停数等关键数字。
- 板块名称。
- 股票名称和代码。
- 涨跌幅、成交额、排名等结构化事实。

失败策略：

1. 自动把错误项反馈给 LLM 重试。
2. 最多自动重试 3 次。
3. 仍失败则标红。
4. 用户可选择重试、使用规则模板或手动编辑。

## 9. 前端交互设计

### 9.1 桌面编辑流程

使用顶部步骤条：

```text
生成报告 → 系统校验 → 人工修正 → 预览长图 → 导出归档
```

### 9.2 编辑权限

- 事实数据不可直接编辑。
- 人工备注、板块排序、标签、新增主线、删除主线可编辑。
- 所有人工动作保存为 `override`。
- 人工修改后可点击“根据修改重写文案”。系统不会自动重写整篇报告。

### 9.3 任务进度

v0.1 展示阶段进度：

- 采集行情。
- 计算评分。
- 搜索新闻。
- 生成文案。
- 事实校验。
- 渲染导出。

详细日志在 v0.3 增强。
