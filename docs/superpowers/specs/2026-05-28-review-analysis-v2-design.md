# 复盘分析维度 v2 设计

## 背景

用户提供的 `2026-05-28_wechat_article.html` 展示了一套更接近人工复盘的结构：先判断市场状态和资金路径，再拆解主线、分支、持续性和次日条件。当前项目虽然已有结构化复盘、次日预测、历史主题回看、资金证据等模块，但分析维度仍偏“强势板块列表 + 通用文案”，缺少对昨日预判、资金轮动、题材阶段、持续性排序和错误修正的系统表达。

本设计目标是把报告从“数据罗列型”升级为“交易复盘型”。HTML 是项目核心产物，因此结构、模块顺序和语言质量优先级高于单纯增加数据源。

## 参考 HTML 的核心框架

参考文件的有效结构如下：

1. **核心结论**
   - 用一句话判断当天市场状态，例如“早盘惯性下探→午后 V 型反转”。
   - 给出最关键资金信号，例如电子行业从净流出逆转为净流入。
   - 对比昨日和今日的主线变化。

2. **指数与市场情绪**
   - 指数表：上证、深成指、创业板、科创 50、北证 50、上证 50。
   - 情绪表：上涨家数、下跌家数、涨停、跌停、成交额，并和昨日对比。
   - 阶段判断：不是只说涨跌，而是归类为退潮、修复、结构性反弹、扩散、分歧等。

3. **昨日预判验证**
   - 逐条列出昨日判断。
   - 标记正确、部分正确、错误。
   - 解释偏差原因，尤其是情绪带来的认知偏差。

4. **板块详细分析**
   - 每个方向包含：方向名、持续性评级、催化、核心标的、梯队/高度、资金、结论。
   - 强方向和弱方向都要写，弱方向用于解释资金流出或一日游。
   - 不只是涨幅排序，要判断“接棒主线、独立逻辑、梯队完整、整体失血、一日游”等阶段。

5. **资金轮动路径**
   - 用路径描述资金从哪里撤出、流向哪里。
   - 识别主线内部轮动，例如“封测/存储 → CPO/MLCC/散热/三代半导体”。
   - 给出关键发现，解释为什么不是全面退潮或为什么是主线扩散。

6. **板块持续性排序**
   - 用持续性排序，而非单日涨幅排序。
   - 排序理由必须包含催化、前排、资金、梯队、风险中的至少两项。

7. **明日操作思路**
   - 分为重点关注、谨慎观察、规避方向。
   - 给出明日重点信号，例如指数是否放量、前排是否承接、后排是否掉队。
   - 明确禁止无证据追高和一日游后排。

## 新分析维度

### 1. 市场阶段维度

新增市场阶段判断，输出到结构化复盘顶部：

- `panic_decline`：恐慌退潮
- `repair`：情绪修复
- `structural_rebound`：结构性反弹
- `mainline_expansion`：主线扩散
- `internal_rotation`：主线内部轮动
- `defensive_rotation`：防御轮动
- `mixed_divergence`：分化震荡

阶段判断的主要依据：

- 指数涨跌和盘中形态，当前 MVP 先用收盘涨跌、创业板/科创强弱、市场标签近似。
- 涨跌家数、涨停跌停数量、成交额变化。
- 当日强方向与昨日强方向是否延续、切换或扩散。
- 复盘源是否确认封板效率、连板高度、主线关键词。

### 2. 昨日预判验证维度

当前已有 `prediction_review`，但内容偏模板化。v2 要把它改为“逐条验证”：

- `claim`：昨日判断原文或摘要。
- `result`：实际结果。
- `verdict`：正确、部分正确、错误。
- `evidence`：指数、板块、个股、资金或复盘源证据。
- `bias_reason`：如果错误，说明误判来自情绪、数据不足、主线理解错误、资金方向误读等。

数据来源优先级：

1. 前一日报告的 `next_day_predictions`、`tomorrow_judgement`、`action_discipline`。
2. 前一日结构化 JSON 快照。
3. 用户提供的参考 HTML 或历史 HTML 解析结果。
4. 如果没有前一日报告，明确写“缺少昨日预判，不做机械验证”。

### 3. 板块阶段维度

每个板块不再只有 `score` 和 `pct_change`，而要有阶段标签：

- `leader`：主线核心
- `new_leader`：接棒新核心
- `branch_expansion`：主线分支扩散
- `independent_theme`：独立逻辑
- `repair_only`：修复观察
- `weakening`：走弱/失血
- `one_day`：一日游
- `avoid`：规避

判断依据：

- 前排个股是否有 20cm/30cm、连板、高度股。
- 是否有多个核心标的共同走强。
- 资金强度：成交额、换手率、主力净流入文本证据。
- 是否被同花顺/东方财富复盘源确认。
- 是否在昨日强方向中延续或掉队。

### 4. 资金轮动维度

新增独立的资金路径模块：

- `previous_path`：昨日或上一阶段资金所在方向。
- `today_path`：今日资金流向。
- `rotation_type`：回流、扩散、切换、防御、一日游撤离。
- `key_signal`：最关键证据。
- `interpretation`：一句话解释对明日的影响。

MVP 不强求全市场逐行业资金流接口，优先从现有数据合成：

- TickFlow 个股成交额、换手率、涨幅。
- Anspire 新闻催化。
- 同花顺/东方财富复盘文本中的资金、连板、高度、题材描述。
- 当前报告和前一日报告的强势方向差异。

### 5. 持续性排序维度

持续性排序独立于强度排序。排序权重建议：

- 前排质量：25%
- 梯队完整度/高度：20%
- 资金强度：20%
- 催化确定性：15%
- 历史延续/主线地位：10%
- 风险扣分：10%

输出要求：

- 至少列出 5 个方向。
- 必须包含强方向、观察方向、规避方向。
- 每个方向给出评级：高、中高、中、中低、低。
- 每个理由必须是具体证据，不允许“综合评分靠前”这类空话。

### 6. 明日策略维度

明日模块分三层：

- **重点关注**：主线核心或新核心，必须有明确前排标的。
- **谨慎观察**：修复、扩散、低位补涨，必须给触发条件。
- **规避方向**：一日游、后排、失血方向，必须说明风险。

每个方向输出：

- 观察标的。
- 触发条件。
- 失效条件。
- 不能做的动作。

## 数据与结构调整

### 保留的数据源优先级

1. TickFlow：行情、涨跌、成交额、换手率、前排个股。
2. Anspire：新闻和催化。
3. 同花顺复盘：主线、封板、高度、题材文字确认。
4. 东方财富涨停复盘：涨停结构和文章线索。
5. 历史报告快照：昨日判断验证和持续性对比。

### 不做的事情

- 不再用硬编码关键词强行生成假主线。
- 不为了“同花顺概念指数”牺牲报告质量。
- 不在数据不足时伪造具体资金流或标的。
- 不把弱板块包装成机会。

### 新增/调整模型建议

在 `StructuredReviewDTO` 附近新增或扩展以下结构：

- `MarketPhaseReview`
  - `phase`
  - `headline`
  - `key_signal`
  - `yesterday_today_compare`

- `PredictionVerificationItem`
  - `claim`
  - `verdict`
  - `actual_result`
  - `evidence`
  - `bias_reason`

- `SectorDeepDive`
  - `sector`
  - `stage`
  - `rating`
  - `catalysts`
  - `core_stocks`
  - `capital_evidence`
  - `team_structure`
  - `conclusion`
  - `watch_signals`
  - `avoid_signals`

- `CapitalRotationReview`
  - `path`
  - `rotation_type`
  - `key_finding`
  - `next_watch`

现有字段可以兼容保留，避免一次性破坏前端模板和 API。

## 报告 HTML 顺序

新 HTML 推荐顺序：

1. Header：日期、报告类型、核心主题、指数/成交/涨停标签。
2. 核心结论。
3. 指数与市场情绪。
4. 昨日预判验证。
5. 板块详细分析。
6. 资金轮动路径。
7. 板块持续性排序。
8. 明日操作思路。
9. 历史强势主题跟踪。
10. 新闻与数据源状态。

## MVP 实施边界

第一版只做“分析框架升级”，不重新切换底层数据源，不再重复上一次失败路径。

MVP 必须达成：

- 报告结构和参考 HTML 一致。
- 每个强板块都有具体标的或明确写证据不足。
- 有昨日预判验证模块。
- 有资金轮动路径模块。
- 有持续性排序模块。
- 明日策略必须分关注、观察、规避。
- 文案不能 fake，不知道就写“证据不足”。

MVP 可以暂缓：

- 全市场行业/概念真实资金净流入接口。
- 同花顺概念指数全量排行。
- 自动识别盘中 V 型形态。
- 更复杂的 LLM 复盘评价。

## 验收标准

1. 用现有 fake provider 的测试报告也能生成完整结构，但必须标明证据有限。
2. 用真实 TickFlow/Anspire/复盘源时，强方向必须包含具体标的。
3. 对上一日报告存在时，必须生成逐条预判验证。
4. HTML 中不能出现空模块或“暂无数据”大面积占位；无证据模块应折叠或用一句明确说明。
5. 生成 PNG 不破版。
6. 不引入生产 fake 内容。

## 推荐实施顺序

1. 新增结构化 schema，兼容旧字段。
2. 新增 `review_analysis_v2` builder，先用当前报告对象和历史报告生成 v2 DTO。
3. 改 HTML 模板顺序和组件，但保留旧数据源。
4. 增加测试覆盖参考 HTML 的关键模块。
5. 再评估是否补充新的真实数据接口。
